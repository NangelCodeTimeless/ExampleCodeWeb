Presentación de diapositivas de desplazamiento infinito totalmente CSS

Autor
Chris Coyier
Última actualización
27 de mayo de 2016

Aprenda Desarrollo en Frontend Masters

Solo por diversión, quería ver si podía hacer que una fila de imágenes se animara en una página y se repitiera indefinidamente. Resulta que en realidad no es tan difícil.

El concepto es mover la posición del gráfico largo

Primero, el gráfico debe diseñarse de manera que se repita una parte de él. Luego, anima la posición izquierda de esa imagen hasta que parece que se ha completado un ciclo completo (pero realmente se mueve lo suficiente como para parecer idéntica), luego vuelve rápidamente a la posición original y comienza de nuevo.

Solo necesitamos un envoltorio externo para estar en una posición relativa y ocultar el desbordamiento, luego un contenedor interno en el que animar (que es la imagen). 

<div class="slideshow">
   <div class="images">
      A slideshow of images in here (whatever you want to say for screen readers)
   </div>
</div>
.slideshow {
  position: relative;
  overflow: hidden;
}
.images {
  background: url(slideshow.jpg);
  position: absolute;
  left: 0;
  top: 0;
  height: 100%;
  width: 300%;
  animation: slideshow 10s linear infinite;
}
@keyframes slideshow {
  0%    { left: 0; }
  100%  { left: -200%; }
}

Saquemos un mejor rendimiento de esta cosa, vaquero.

El código anterior debería funcionar bien, pero puede ser un poco irregular dependiendo del navegador / plataforma / versión / recursos disponibles. Podemos obtener un mejor rendimiento (al menos en WebKit / Mobile WebKit) al decirle al navegador que vamos a utilizar transformaciones 3D (aunque no lo hagamos) y luego usarlas en translateXlugar de hacerlo left.

NOTA: Este truco aumenta específicamente el rendimiento (que normalmente no obtendrías) de las transformaciones 2D y los cambios de opacidad. Si no estuviéramos cambiando a usar translateX, no veríamos ningún beneficio.

.images {
  ...
   
  /* Hey browser, use your GPU */
  transform: translate3d(0, 0, 0);
}

@keyframes moveSlideshow {  
  100% { 
    -webkit-transform: translateX(-200%);  
  }
}

No soy un gran fanático del hecho de que esencialmente necesitamos engañar al navegador para que tenga un mejor rendimiento, pero como personas de CSS, supongo que estamos acostumbrados a ese tipo de cosas. Creo que la forma original en que lo escribimos está bien y el navegador debería reconocer cuándo puede ser más eficiente y hacerlo.

Agregar fantasías

Con mi presentación de diapositivas, quería hacer dos efectos geniales adicionales más allá del desplazamiento de las imágenes. Cuando pasaste la presentación de diapositivas, sería:

  1. Acelerar
  2. Cambiar de blanco y negro a color

Para acelerarlo, podrías pensar, ¡solo reduciremos la duración al pasar el mouse! Y por lo tanto:

.slideshow:hover .images {
  animation-duration: 5s;
}

¡Pero esto va a ser desigual! Al cambiar la duración de la animación, crea una nueva línea de tiempo de las propiedades que deben tener, con qué valores y cuándo. Descubre esos nuevos valores y salta a donde deberían estar inmediatamente. Que no sólo “acelerar” de su exacta posición actual. Me temo que no conozco una forma realmente sólida de evitar esto.

En cambio, solo ajustaremos el concepto de "acelerar". Si configuramos dos presentaciones de diapositivas una encima de la otra, cada una de ellas corriendo a diferentes velocidades, y luego ocultamos / mostramos la ejecución más rápida en la posición de desplazamiento, podemos obtener un efecto de "aceleración" bastante decente. Usaremos transiciones para hacer la transición más suave.

<div class="slideshow">
   <div class="images-1">
     A slideshow of images in here (whatever you want to say for screen readers)
   </div>
   <div class="images-2">
   </div>
</div>
.slideshow > div {
   ...

   transition: opacity 0.5s ease-out; 
    
   /* Slow */
   animation: moveSlideshow 60s linear infinite;
    
   ...
}

.images-1 {
  /* Fast */
  animation: moveSlideshow 20s linear infinite;
}

.slideshow:hover .images-2 {
   opacity: 0;
}

Para lo blanco y negro, lo que haremos será la presentación de diapositivas más alta en blanco y negro y el color de presentación de diapositivas más bajo / más rápido. Para guardar una solicitud HTTP, haremos que ambos tengan la misma imagen y solo cambiaremos la posición de fondo.

.images-1 {
  /* Sprite */
  background-position: 0 200px;

  ...
}

¡Entonces eso debería hacerlo!

Origen

Esta idea surgió en la página de eventos de Wufoo Hearts Tech que hicimos recientemente, donde quería mostrar imágenes de varios de los eventos en los que hemos estado recientemente. Y, lo admito, solo quería jugar con esta idea.

      Esa es una discusión interesante para comenzar.

      Yo diría que el otro lado, en que CSS es para cosas como esta y JS no lo es.

      In the classic conversation of “separation of concerns”, CSS is for design/styling and JS is for behavior. An animation is, if you ask me, far more a part of design/styling than it is behavior. It depends on the use of the animation though. A slideshow is pretty gray-area since there is plenty of behavior that can go along with them.

      The other thing to discuss is HOW the animation is accomplished. To animate something with JavaScript, you need to rapidly manipulate inline styles on an element. I’d say this is closer to a clever hack than an appropriate way to handle animation. As shown in this tutorial, by making animation happen in CSS, we can actually get the GPU (graphics processor unit) involved and have it handle the animation. That is almost inarguably more efficient than JavaScript =)
